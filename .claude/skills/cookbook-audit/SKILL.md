# Cookbook Whole-Product Audit

Комплексный аудит кукбука по пяти линзам сразу. Не только код — **продукт целиком**: UX, UI, продуктовая логика, продакт-менеджмент и техника.

## Когда использовать
Пользователь просит проанализировать существующую функциональность кукбука, понять «что и как работает», найти слабые места и сделать выводы. Это про приложение `landing/` + `api/routes/cookbook.py` (личная поваренная книга), НЕ про фешн-бота.

## Как запускать
1. **Параллельно** запусти пять сабагентов (в одном сообщении, несколько Agent-вызовов) — каждый со своей линзой:
   - `cookbook-ux-auditor` — потоки/трение/discoverability
   - `cookbook-ui-auditor` — визуал/вёрстка/консистентность
   - `cookbook-logic-auditor` — осмысленность правил фич (поведенческие баги)
   - `cookbook-pm-auditor` — позиционирование/ров/retention/приоритеты (читает память и Анализ №4)
   - `cookbook-code-auditor` — мёртвый код, связки, стоимость моделей, покрытие тестами
   Дай каждому один и тот же контекст: корень `/home/stas/fashion-bot`, ключевые файлы (`landing/js/app.js`, `landing/js/assistant.js`, `landing/js/sync.js`, `landing/index.html`, `landing/sw.js`, `api/routes/cookbook.py`, `worker/tasks/cookbook_push.py`), `CLAUDE.md` и память `~/.claude/projects/-home-stas-fashion-bot/memory/`.
2. Пока агенты работают — прогони быстрые техпроверки сам (или вызови скил `cookbook-check`): `node --check landing/js/*.js landing/sw.js`, grep-контракты (SYNC_KEYS в трёх файлах, `sw.js VERSION` ↔ `index.html ?v=`).
3. Собери находки всех линз. Дедуп по смыслу.

## Сведение и приоритизация
Сформируй **единый отчёт**:
- Раздел на каждую линзу (UX / UI / Логика / PM / Техника) — топ-находки с `severity × effort`.
- Сквозной **топ приоритетов**: отсортируй все находки по (severity, обратный effort). Разведи «баги» (ломает/жжёт деньги/дыра/путает юзера) и «улучшения».
- **Известные семена** (проверь, что не потерялись): мёртвый ключ `planServings`, мёртвая константа `HAIKU` (`cookbook.py`), `/import` по фото на платном Sonnet вместо бесплатной llava, ноль e2e-тестов на эндпоинты.
- **Выводы и следующие шаги**: 5–8 пунктов «что чинить/строить первым» с обоснованием.

## Правила
- Каждая находка — с `file:line` и доказательством из кода/памяти, а не догадка.
- Ничего не чинь в этом прогоне — только диагностика и выводы (если пользователь не попросил фиксить).
- Масштабируй под запрос: «бегло» — по одной ключевой находке на линзу; «тщательно/полный аудит» — полный список + сквозная приоритизация.
