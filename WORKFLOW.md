# Fashion Bot — WORKFLOW.md

## Подход к разработке: защитное проектирование перед делегированием

Claude Code — мощный но слепой исполнитель. Он сделает ровно то что написано
в промте, даже если это сломает три других вещи. Этот документ описывает как
думать за него до того как он начал писать код.

---

## 7 линз анализа

Перед каждым промтом в Claude Code смотрим на задачу через 7 линз:

**1. Логика и архитектура**
- Связи между компонентами, edge cases
- Циклические импорты, корректность async/await
- Атомарность DB операций (AsyncWriteSession/AsyncReadSession)
- Конфликты с существующим кодом

**2. Тестируемость**
- Что Claude Code проверит автоматически (pytest)
- Что требует глаз Стаса (визуальные СТОП точки)
- Конкретные критерии СТОП: "написать результат"

**3. Деплой и синхронизация**
- Какие контейнеры рестартовать
- Worker sync (morning_brief, style_config, permissions — обязательно!)
- Нужна ли Alembic миграция
- Порядок операций при деплое

**4. UX и текст**
- Формулировки сообщений
- Edge cases: пустой гардероб, None, новый пользователь без вещей
- Тон: Касси — дружелюбный стилист, не официально
- Что видит пользователь при ошибке

**5. Безопасность и экономия**
- Токены Claude API (модели и лимиты — см. CLAUDE.md раздел "Стек")
- Rate limits, защита от злоупотреблений
- None значения, неожиданные входные данные
- Admin проверка для тестовых команд

**6. Откат**
- Что делать если сломалось в продакшне через час
- `git revert` для кода, SQL для данных если нужно
- Docker: `docker restart` или откат на предыдущий коммит
- Как проверить что откат прошёл успешно

**7. Наблюдаемость**
- Какие structlog строки появятся в логах при успехе
- Что проверить через `docker logs docker-app-1` через 5 мин после деплоя
- Как отличить "работает" от "молча сломано"

## Режимы анализа

### Тривиальный фикс (1-2 строки, один файл)
Прогон не нужен. Просто проверь: не ломает ли импорты, нужен ли worker sync.

### 5 итераций (быстрый режим)

Для большинства задач — прогон через все 7 линз за 5 итераций:

```
Итерация 1: Логика + архитектура → саммари проблем
Итерация 2: Тестируемость + наблюдаемость → саммари проблем
Итерация 3: Деплой + синхронизация + откат → саммари проблем
Итерация 4: UX + текст → саммари проблем
Итерация 5: Безопасность + экономия → саммари проблем
→ Финальный промт для Claude Code
```

> Все 7 линз распределены по 5 итерациям: наблюдаемость объединена с тестируемостью,
> откат — с деплоем. Ни одна линза не пропускается.

### Three-Pass анализ (для сложных задач)

Для батчей 5+ файлов или рискованных изменений (платежи, миграции, auth, онбординг):

```
PASS 1 — РИСКИ (7 линз)
Прогон через все 7 линз → список проблем 🔴/🟡/🟢

PASS 2 — УТОЧНЕНИЕ
- Какие файлы затронуты кроме очевидных?
- Скрытые зависимости между проблемами?
- Что из Pass 1 — перестраховка, а не реальная проблема?
→ Уточнённый список: что реально важно

PASS 3 — ВЗГЛЯД ДЖУНА
Прочитай задачу глазами исполнителя:
- Что непонятно? Где можно интерпретировать не так?
- Какие СТОП точки не покрыты автотестами?
→ Что добавить в промт для однозначности
```

**Саммари:**
```
🔴 БЛОКЕРЫ: ...
🟡 ВАЖНО: ...
🟢 МОЖНО ПРОПУСТИТЬ: ...
🎯 СТОП ТОЧКИ: [тест + критерий успеха]
```

## Когда использовать какой режим

| Задача | Режим |
|--------|-------|
| Мелкий фикс (1-2 строки) | Тривиальный |
| Фича в 1-2 файлах | 5 итераций |
| Средняя фича (3-5 файлов) | 5 итераций + СТОП точки |
| Большой батч (5+ файлов) | Three-Pass |
| Платежи / миграции / auth | Three-Pass обязательно |
| Онбординг / конверсия | Three-Pass |

## Структура финального промта для Claude Code

Каждый промт должен содержать:

```
ШАГ 0: Диагностика
  Конкретные команды чтобы понять текущее состояние
  git log --oneline -10, проверка текущего кода

ЗАДАЧА 1: [название]
  Файл: конкретный путь
  Что менять: конкретный код
  Почему: объяснение

АВТОТЕСТЫ
  docker exec docker-app-1 python3 -m pytest ...
  Ожидаемый результат: PASS

СТОП точки (визуальные)
  *** СТОП 1 — описание ***
  Критерии: что должно быть
  Написать мне результат.

ФИНАЛ
  git add && git commit && git push
  Worker sync если нужен (см. CLAUDE.md раздел "Деплой")
```

## Красные флаги (стоп и думай)

- Меняется `permissions.py` → проверить все places где используется
- Меняется `morning_brief.py` → worker sync обязателен
- Новые поля в моделях → Alembic миграция первым делом
- Меняется `/start` или onboarding → тест с новым пользователем
- Платёжный флоу → test payload `test:` не должен активировать premium
- Admin функции → двойная проверка `user.plan == "admin"`
- Меняется `scheduler.py` → проверить cron-выражения, timezone
- Меняется `redis.py` или `queue.py` → проверить все consumers
- Меняется `docker-compose.yml` → memory limits, volumes, networks
- Меняется `config.py` → проверить .env в контейнерах
